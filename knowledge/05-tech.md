# 搭+ 技术决策与架构

## 技术栈

| 层 | 技术选型 |
|----|---------|
| 前端 | 微信小程序原生开发 |
| 后端API | RESTful |
| 数据库 | MySQL (关系型) |
| 支付 | 拉卡拉分账模式（用户→中间账户冻结→分账至主办方+平台→提现） |
| 通知 | 小程序内服务通知中心 + 微信模板消息(关键节点) |

## 架构模式

分层架构: 前端层 → 应用层/API → 领域服务层 → 数据访问层

建议将「报名人管理」作为独立微服务模块。

## 6 个关键架构决策

### 1. 零审核策略（从人工审核演进而来）

V1.0 PRD定义了完整的人工审核流程(含敏感词检测)，后演进为零审核「即发即上」模式。未来通过评价数据实现「后置审核」。

**理由**: 降低发布摩擦，提升B端体验。

### 2. 比赛独立业务线

比赛不是扩展普通活动，而是创建独立的业务线。

**理由**: 总体适配度仅40%深度定制需求高、避免与普通活动混淆、不影响现有业务风险可控、共享基础架构但拥有独立专业功能。

### 3. 拉卡拉分账模式

采用拉卡拉分账能力实现平台资金流转：用户支付→资金冻结在拉卡拉中间账户→活动结束后平台发起分账指令→主办方约90%、平台约10%分别入账→各自提现。交易手续费2.5‰由平台承担。分账完成时点确认平台技术服务费收入（增值税6%）。退款采用「先分账后冲正」流程。

**风险点**: 商户进件流程复杂（6+步），中小主理人入驻转化率低，制约交易规模增长。

### 4. 区域筛选按行政区划而非车程/距离

用户曾提出按车程/时间成本筛选，最终坚持按县区行政区划。

**理由**: 手中有县区资源。

### 5. 目的地指南轻量化

采用「三明治模板」架构(元数据层+富文本层+动态组件层)，放弃复杂POI数据库和地理位置引擎。

### 6. 排行榜T-30滚动窗口

每日凌晨00:00重新计算，仅统计过去30天数据，避免阶层固化。前端展示分=真实热度+虚拟热度加成。

## 敏感信息处理规范

- **身份证号**: AES加密存储，后台查看脱敏(`110101********1234`)
- **API**: 默认不返回id_card，需权限才可解密
- **日志**: 禁止打印完整身份证号
- **字段命名**: 统一使用 name/phone/id_card (禁止 real_name/mobile/identity/cert_no 等变体)

## 数据安全

- 主办方扫码需账号权限认证
- 后端强制权限校验，防止前端绕过
- 银行卡数据加密存储
- 二维码含加密token，唯一防伪，有时效性

## 兼容性

- 适配主流微信小程序运行环境(iOS和Android各版本微信)
- 兼容主流手机操作系统版本
